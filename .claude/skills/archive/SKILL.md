# 归档讨论到知识库

## 元数据

| 项目 | 内容 |
|------|------|
| 用途 | 提炼对话要点 → 定位知识树节点 → 用户确认 → 追加写入 JSON |
| 输入 | 当前对话内容；`workspaces/<kb_id>/knowledge_tree.json` |
| 输出 | 确认消息（归档路径 + 写入内容摘要）|
| 副作用 | **追加写** `workspaces/<kb_id>/knowledge_tree.json` 中节点的 `summary` 字段 |
| 需要确认 | 是 — 展示归档预览后等用户确认 |

## 触发条件

**会触发：**
- "归档一下"
- "保存到知识库"
- "把这些存进去"
- "记录下来"
- "加到知识库里"

**不触发：**
- "帮我解释一下 GRPO" → ask-kb
- "同步到飞书" → sync-wiki
- "规划知识库" → plan-wiki
- "记录面试题" → interview

## 执行流程

### 第一步：提炼知识要点

回顾当前对话，仅保留值得长期存储的内容：
- 核心概念和定义
- 关键结论和技术对比
- 来自联网搜索的新信息

**去掉**：闲聊、提问过程、重复描述、上下文铺垫。

提炼结果控制在 3–8 条要点，每条不超过两句话。

### 第二步：定位目标节点

读取 `workspaces/<kb_id>/knowledge_tree.json`，判断内容应归入哪个节点：

| 情况 | 处理方式 |
|------|---------|
| 匹配现有节点 | 追加到该节点的 `summary` |
| 需要新的子节点 | 在合适父节点下创建新节点（直接编辑 JSON）|
| 需要顶层新分类 | 建议用户确认后添加到根节点 |

### 第三步：预览并确认

向用户展示归档预览，**等待确认后才写入**：

```
📥 归档预览

目标节点：知识库 > 模型架构 > MoE
操作方式：追加到现有 summary

归档内容：
  - MoE 路由机制：使用 softmax + TopK 选择激活专家，训练时加辅助 load balancing loss
  - 稀疏激活：每次前向传播只有 Top-K 个专家参与计算，显著降低推理 FLOPs

---
确认归档？（输入"确认"继续，或说明需要修改的内容）
```

### 第四步：执行归档

用户确认后运行：

```bash
# 默认 workspace
python ${CLAUDE_SKILL_DIR}/scripts/archive_to_kb.py "<节点路径>" "<归档内容>"

# 指定 workspace
python ${CLAUDE_SKILL_DIR}/scripts/archive_to_kb.py --kb <kb_id> "<节点路径>" "<归档内容>"
```

路径用 `>` 分隔，如 `"模型架构>MoE"`。

如需新建节点（脚本不支持自动创建），直接编辑 `workspaces/<kb_id>/knowledge_tree.json`。

### 第五步：归档后提示

- 确认归档成功，显示写入的节点路径
- 如果该节点已有 `obj_token`（说明已同步到飞书），提醒：
  > "该节点已同步到飞书，本次修改仅在本地生效。如需同步，运行 /sync-wiki。"
- 如果是新建节点，提醒：
  > "新节点尚未同步到飞书，如需同步请运行 /sync-wiki。"

## 归档内容写入格式

追加到节点 `summary` 时使用以下格式：

```
[原有内容（如有）]

---
[YYYY-MM-DD] 补充：
- 要点 1
- 要点 2
```

## 错误处理

| 错误 | 处理方式 |
|------|---------|
| `knowledge_tree.json` 不存在 | 提示先运行 plan-wiki 建立知识树 |
| 节点路径不存在 | 显示可用顶层节点列表；建议用户确认路径或新建节点 |
| 内容为空 | 要求用户指定要归档的内容 |
